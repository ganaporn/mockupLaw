OLD Suggestion 1: T_CASE — DEFENDANT_NAME เป็น String เดียว

คดีบางคดีมีหลายจำเลย เช่น คดีอุบัติเหตุที่มีทั้งคนขับรถและเจ้าของรถเป็นจำเลย ปัจจุบัน DEFENDANT_NAME รับได้แค่คนเดียว ควรพิจารณาเพิ่ม T_CASE_PARTY (CASE_ID, PARTY_ROLE_CODE, PARTY_NAME, ID_CARD_NO, ADDRESS) เพื่อรองรับหลายคู่ความ

OK แต่แนะนำ Suggestion 2: T_CONTRACT_DEBT — ขาด IS_LAWSUIT_APPROVED

Flow เตรียมคดีมี "เสนอขออนุมัติไม่ฟ้อง" (กรณีหนี้น้อย) และ "เสนอขออนุมัติงดบังคับคดี" (ครบอายุความ 10 ปี) ควรเพิ่ม APPROVAL_TYPE_CODE (NO_LAWSUIT / STOP_ENFORCEMENT), APPROVAL_DATE, APPROVER_USER_ID ใน T_CONTRACT_DEBT หรือ T_CASE_ACTIVITY

OK แต่แนะนำ Suggestion 3: M_CASE_STATUS — ควรมี ALLOWED_NEXT_STATUS list

เพื่อ validate state machine ใน backend แนะนำเพิ่ม table M_STATUS_TRANSITION (FROM_STATUS_CODE, TO_STATUS_CODE, REQUIRED_ROLE_CODE) แทนการ hardcode ใน code — ทำให้ Admin ปรับ flow ได้โดยไม่ต้อง deploy ใหม่

OK แต่แนะนำ Suggestion 4: M_USER — ขาด M_USER_ROLE (many-to-many)

ปัจจุบัน M_USER มี ROLE_ID เดียว (FK) ทำให้ User มีได้แค่ 1 Role ในทางปฏิบัติผู้ใช้อาจมีหลาย Role เช่น เป็นทั้งนิติกรและ admin ควรใช้ M_USER_ROLE (USER_ID, ROLE_ID) แทน

OK แต่แนะนำ Suggestion 5: File Storage Strategy

FILE_PATH เป็น string ใน T_CASE_DOCUMENT และ T_CASE_REQUEST_DOCUMENT ควร document ว่าเป็น path แบบใด (relative / absolute / object storage URL) และควรมี policy เรื่อง file retention และ backup โดยเฉพาะเอกสารคดีที่ต้องเก็บ 10+ ปี

OK แต่แนะนำ Suggestion 6: T_CASE — ขาด DEADLINE_DATE / STATUTE_OF_LIMITATIONS_DATE

DB Design v3 ✅ ครบถ้วน 100% — แก้ Gap ทั้ง 4 รายการ และ Suggestion ทั้ง 6 ข้อครบแล้ว เพิ่มตารางใหม่ 7 ตาราง บันทึก field ใหม่ 6 columns ในตารางเดิม รวม Schema มี 33 ตาราง

สิ่งที่เพิ่ม/แก้ใน v3:

  1. ✅ T_CASE_PAYMENT — ประวัติการชำระเงินรายงวด
  2. ✅ T_COURT_PROCEEDING — รายละเอียดชั้นศาล (ชั้นต้น/อุทธรณ์/ฎีกา)
  3. ✅ T_NOTIFICATION — ระบบแจ้งเตือน + IS_READ
  4. ✅ T_CASE_PARTY — หลายคู่ความต่อคดี
  5. ✅ M_STATUS_TRANSITION — Configurable workflow state machine
  6. ✅ M_ACTION_PERMISSION — RBAC table
  7. ✅ M_USER_ROLE — Many-to-many user ↔ role
  8. ✅ T_CASE: +STATUTE_LIMITATION_DATE, +ENFORCEMENT_EXPIRY_DATE
  9. ✅ T_CONTRACT_DEBT: +APPROVAL_TYPE_CODE, +APPROVAL_DATE, +APPROVER_USER_ID
  10. ✅ T_LITIGATION_DETAIL: +COURT_LEVEL_CODE, +APPEAL_FILED_DATE, +APPEAL_JUDGEMENT_DATE, +APPEAL_RESULT_CODE

DB Design ปัจจุบัน (v2) ถือว่า ✅ ครอบคลุมดีมาก รองรับ Flow หลักทุกขั้นตอนได้ครบ ทั้ง 3 ส่วนงาน (เตรียมคดี, งานคดี, บังคับคดี) รวมถึง Flow พิเศษอย่าง Migrate data, Auto-create จากระบบอุบัติเหตุ, และคดีล้มละลาย
